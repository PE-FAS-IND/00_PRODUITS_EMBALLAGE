Introduction

Objectif

L'objectif principal de ce software d'emballage est de remplacer les 2 softwares d'emballage (1 pour les FLATPROP + 1 pour le MEDIC).
Les données d'emballage sont enregistrées dans une table BDD unique (dbo.PRODUIT_EMBALLAGE).
Une vérification de tous les tests effectués sur le produit est faite avant d'emballer à fin d'assurer un produit conforme.

Document

Ce fichier sert d'aide pour le software d'emballage BDD. Il est découpé en 3 parties.
La 1ère partie s'adresse plus directement à la production:

  • le première chapitre décrit les différentes onglets de la face avant.
  • Un autre chapitre explique comment créer une commande interne, comment reprendre une commande où le mélange des lots n'est pas possible.

La 2ème partie s'adresse plus aux dévelopeurs:

  • Un chapitre est consacré aux tables BDD. Une table pour l'enregistrement des données et une autre qui sert à la vérification des produits.
  • Un autre chapitre démystifie le chargement des paramètres d'emballage pour le programme.
  • Un autre chapitre explique comment rajouter une imprimante

La dernière partie donne des informations supplémentaires:

  • Un chapitre concerne les divers (par exemple: Où trouver plus d'infos?, paramétrer le lecteur).
  • L'historique des versions est dans un autre chapitre.

Face avant Software (Onglets)

Initialisation

Plusieurs informations sont visibles dans l'onglet initialisation:

  1. Nom du PC
  2. Etat: indique dans quel état est le software
  3. Etat initialisation: LED ON => initialisation réussit
  4. Etat de la sécurité: LED OFF => mise en sécurité
  5. Durant le software des informations sont envoyé à la console
    6. (exemple: synchronisation des tables, commentaire sur erreur, ...)
  6. Indicateur d'erreur
  7. Bouton d'aide (ouvre ce document)

Production

Le programme d'emballage s'ouvre dans cette onglet.
La fonction principale de ce programme est de vérifier si les valves/blocs ont été testé bonne et enregsitrer les données d'emballage en base de donnée.
Avant toute chose un vi va apparaître. Ce vi sert à choisir le produit qu'on veut emballer:


Il y a aussi des fonctions secondaire. Ils sont accessible que pendant la période où on scanne les pièces pour emballer. Ces fonctions sont énumérés ci-dessous:

  1. imprimer les n° d'une commande
  2. Afficher le contenu d'un carton
  3. Pièces en cours d'emballage
  4. Pièces déjà emballées (historique)
  5. Quitter

Imprimer les n° d'une commande

Cette fonction énumère les pièces d'une commande après avoir saisi un n° de commande et valider avec "OK".
Il est possible d'imprimer cette liste et de supprimer la commande dans la base de donnée SQL.
Cette dernière option (supprimer) est seulement disponible aux responsables.

Afficher le contenu d'un carton

Cette fonction énumère les pièces dans chaque carton en fonction d'une commande.
Un responsable peut réimprimer une étiquette d'un carton.
Il est possible de remplacer une valve ou même de la supprimer en base de donnée. Cette option est vérouillée par un mot de passe.

Remplacer Valve

Ce vi permet la suppression ou le remplacement d'une valve dans une commande.
Ci-dessous, une image illlustrant le vi "remplacer 1 valve"

Pour supprimer ou remplacer une valve, il faut que la commande ne soit pas bouclé dans la GPAO.
C'est-à-dire que la commande est bouclé lorsque la case "contrôle" de la commande est coché dans la GPAO (Voir image ci-dessous).
Il n'y a pas de garde fou! Aucune vérification de test est faite sur la valve remplaçante

Pièces en cours d'emballage

Cette fonction énumère les pièces en cours d'emballage.
En entrant un No de pièce valide dans le champs "Sélectionner l'élément", la pièce sera mise en évidence dans la liste.

Pièces déjà emballées (historique)

Cette fonction énumère toutes les pièces déjà emballées du produit sélectionné.
En entrant un No de pièce valide dans le champs "Entrez le No de la pièce ...", la pièce sera mise en évidence dans la liste.

Quitter

Cette fonction arrête la série. Le programme se mettra en mode "Safe".

Safe

Le mode Safe est actif lorsque le programme est mise en sécurité.
Depuis ce mode, il est possible :

  1. De recommencer l'initialisation
  2. D'aller dans l'onglet de maintenance
  3. De quitter le programme

Maintenance

Le mode maintenance est protégé par un mot de passe.
Dans cet onglet, il est possible de:

  1. Quitter le mode maintenance (pour aller en mode Safe)
  2. Lire la table bdd local
  3. Initialiser la table hardware
  4. Rechercher le Port Com du lecteur de code barre
  5. Choisir l'imprimante par défaut
    6. (choisir l'imprimante dans la liste déroulant avant de cliquer sur le bouton)
  6. Choisir le décalage à droite par défaut de l'imprimante

Vous constatez que dans ce mode, il y aun onglet qui comporte 2 page:

  • Valeur table bdd: pour la lecture des données de la table bdd local
  • Initialisation: Pour initialiser les différents paramètres (imprimante, port com, ...)

Commandes

La liste des commandes contient les commandes de la GPAO qui n'ont pas encore été facturée et qui n'apparaissement pas dans l'historique des commandes.

Pour la recherche des commandes, on cherche en fonctions de plusieurs champs. Ces champs change en fonction du type de l'article:

  • Pour les articles qui commencent par "12"(FLATPROP) et "17"(FASPROP)
    • NO_ARTICLE
    • NO_FILE
    • DESIGNATION
  • Pour les articles qui commencent par "01"(Microsol) et "04"(Bacosol)
    • NO_ARTICLE
    • NO_FILE
  • Pour les autres articles
    • NO_ARTICLE

Commande interne

Il est possible de créer une commande interne pour tous les produits. Cette commande est enregistrée dans la table des données d'emballage mais nullement dans la GPAO.
Le numéro de la commande interne suit la loi suivant: JJMM/AA-hh.mm.ss
Il n'est pas nécessaire d'entrer la quantité d'une commande interne. Pour arrêter la série, il suffit de scanner 2 fois la même pièce ou de quitter la série (bouton quitter).
A la fin de la série, le software vous indique le nombre de pièces dans la commande interne créée précédemment.
Aucune étiquette d'emballage est crée pour les commandes internes.

Mélange des lots pas possible

Pour certains produits (comme la F120078), il n'est pas autorisée de mélanger les lots lors de la livraison.
Pour ces produits, il faut préciser les différentes lots constituant la commande.
Ci-dessous, la vu du vi où l'on précise l'année, la semaine et la quantité des lots.

Reprendre commande

Il se peut que la commande ne soit pas/ne peut pas être emballé d'un coup. Nous pouvons donc reprendre la commande mais attention aux recommendations suivantes:

  • Le nombre de cartons peut varier en fonction des lots (quantité par lots, nombre de lots).
  • Il ne sera pas possible de continuer un lot plus vieux si un lot plus récent à été emballer.
    • Exemple: Vous avez emballé le lot 1841 ensuite vous commencez à emballer le lot 1839. Vous arrêtez la série et vous la reprenez, il sera impossible de continuer le lots 1839.
  • Si vous ne savez pas la quantité du lot, mettez la valeur maximum.
    • Il sera possible de scanner l'étiquette du dernier carton lors de la reprise de la commande.
  • Pour continuer un lot, il suffit d'entrer la quantité restante
    • Ci-dessous, un exemple pour continuer un lot.

Chargement des Paramètres d'emballage

Le chargement des paramètres se fait obligatoirement pendant l'inialisation.
Il y a la possibilité de forcer la synchronisation lors du choix du produit par le bouton "Synchroniser Paramètres" en bas à droite.
Ce bouton clignote si la synchronisation des paramètres n'a pas eu lieu dans les 23 dernières heures.

Le chargement des paramètres d'emballage se fait de plusieures manières.
Ci-dessous, vous trouvez un petit schéma expliquant les différentes manières de charger les données:

Pour les FLATPROP, il va chercher une partie des informations dans les fichiers "_PP.xml" sur le réseaux au chemin "U:\MTD_PRD_TESTS\PARAM_TESTS\12".D'autres informations sont en dur dans le programme ou tiré de la GPAO.
Pour les Autres produits, il va chercher une partie des informations dans une base de donnée sqlite (nommé Param_Produit) sur le réseaux au chemin "U:\MTD_PRD_TESTS\PARAM_TESTS\EMBALLAGE". Cette base de données est créer manuellement.
Le reste des informations est tiré de la GPAO.

Lors du choix du produit par scannage, si plusieurs produits ont le même code barre, il sera demandé à l'utilisateur de choisir entre ces produits.
(exemple: S122.0008 et S121.0017, le S122.0008 a comme base le S121.0017.)

Comment remplir la table SQLite "param_produit"

Pour des produits uniques qui ont une étiquette, il suffit de remplir les champs correctements:

  • No_Valve_FAS: No Artcile de la GPAO
  • suffixe: valeur du champ "SUFFIXE" de la GPAO
    • Si valeur commence par "champ_", le programme va chercher la valeur du champ après "champ_"
    (par exemple: pour la valve S112.1065, le suffixe sera la valeur du champ "REVIS_SUFF" de la GPAO)
  • designation: désignation de la GPAO
  • indice_produit: indice produit (utiliser dans le code barre)
  • code_barre_fixe: partie fixe du code barre
  • code_barre_fixe_fin: partie final du code barre fixe
    • le code barre fixe se compose de 3 parties:
    code_barre_fin + indice_produit + code_barre_fixe_fin
    Elle n'est pas prise en compte lors du choix du produit par scannage.
  • affiche_code_barre_client: Est-ce que le code barre du client est imprimé sur l'étiquette d'emballage?
  • melange_des_lots_possible: Est-ce que le mélange des lots est possible?
    • Si ce n'est pas possible, le programme va demander à trier les lots.
  • Famille: correspond à la valeur mise dans la table "dbo.CORRESPONDANCE_TABLE"
    • Il permet de faire la correspondance entre le produit et ses tables.
  • date: configuration de la date dans le code barre
    • Ci-dessous, un tableau explique la codification du champs "date":
    Code Description
    A Année
    M Mois
    S Semaine
    Q Quantité livrée
    I "Index"
  • longeur_no_serie: le nombre de dernière chiffre qui vont être imprimer sur l'étiquette
    • Si ce champ n'est pas remplis, la valeur par défaut est 0.

Les sous-chapitres suivants explique les différentes variantes pour remplir la table SQLite "param_produit".

Type de valve

Il est possible qu'on veuille rajouter un type de valve (par exemple: FASPROP "17_2").
Pour cela, il suffit de mettre dans le champs "No_Valve_FAS" une partie du No_Article suivi d'un "%" (exemple: 17_2%).

Pour les types de produit, il se peut que les valeurs des différents champs soient tirées de la GPAO.
Pour cela, il suffit d'écrire dans le champ la valeur suivant : "champ_NomDuChampOuOnTireLInfo"
(Par exemple pour FASPROP, la valeur du champ "designation" est tirée du champ "CLC_DESIGNATION" de la GPAO)

Il se peut aussi que la valeur du champ "code_barre_fixe_fin" soit tirée d'une colonne d'un fichier .csv.
Pour cela, il suffit d'écrire dans le champ la valeur suivant : "fichier_CheminDuFichier_champ_NomDuChampOuOnTireLInfo"
(Par exemple pour FASPROP, la valeur du champ "code_barre_fixe_fin" est tirée de la colonne "autre" du fichier "U:\MTD_PRD_TESTS\PARAM_TESTS\17\Marquage Laser Cheval\FASPROP_PARAMETRES.csv")
La liaison entre les valeurs du fichier .csv et le produit se fait avec la colonne "index" et l'id_article GPAO.

Les champs "indice_produit et "code_barre_fixe, il est possible de rajouter un séparateur.
Pour cela, il suffit de séparer le nom du champ ("champ_NomDuChampOuOnTireLInfo") et le séparateur (;) par un point-virgule(;).
(Par exemple pour FASPROP, le code barre est un 2D qui suit la règle suivante: "ID_ARTICLE;INDICE_GPAO(numérique);DATECODE(AASSIIII);Facultatif")
Pour le champ "indice_produit, la valeur retournée par la GPAO sera transformé en numérique (a = 1, ... , z = 26).

Pas d'étiquette

Vous avez surement constaté que pour les CHIPROP, le champ "code_barre_fixe" contient la valeur "NoLabel".
Les CHIPROP ont une table de données mais pas d'étiquette.
Il n'y a pas de test unitaire fait sur chaque pièce donc pas de vérification unitaire.
Avec la valeur "NoLabel", le produit n'apparait pas pour l'emballage.

Pas de test

Certains produits ne font pas de test (ou les données de test ne sont pas enregistrées en BDD) mais comporte une étiquette pour la traçabilité à l'emballage.
Pour identifier ces produits, la valeur du champ Famille doit commencer par "NoTest".
(Exemple: 04-341E-11-11 F04313 4212 SDGNUL 24V DC 10W)
La possibilité d'ajouter quelque chose après "NoTest" est utile pour la Table sqlite "SNPRODUIT_2D".

Table local SNPROCDUIT_2D

La table sqlite "SNPRODUIT_2D est utile pour les codes 2D (Par exemple: pour FASPROP).

Chargement des Paramètres pour les certificats de conformité

Plusieurs certificats sont envoyées au client. Ces certificats sont soit envoyé par e-mail ou directement mis dans le carton.
Pour permettre la conformité du certificats, des tables SQLite ont été crées. Ce chapitre décrit les différentes tables.

Table local SerialNumbers_Draeger

Une à plusieurs fois par semaine, nous devons envoyer à "IMI FAS Kundenbetreuung" un récapitulatif des envoyées de certains produits.
Cette table SQLite sert à lister les produits concernées. Elle comporte 2 champs: la référence et l'ID_ARTICLE de la GPAO.

Table local NumSerie_Specifique

Il se peut que pour un produit le client veut un fichier spécifique.
Cette table SQLite sert à lister les produits concernées ainsi que leurs paramètres. Elle comporte 5 champs: reference, ID_ARTICLE, fichier_modele, chemin_Enr et imprimante:

Tableau expliquant les différents champs de la table "NumSerie_Specifique"
Champs Description Remarque
référence Nom de la référence du produit Ce champ n'est pas utilisé dans le software.
Il sert uniquement à la comprehension pour le programmeur, l'utilisateur
ID_ARTICLE L'id article de la GPAO Le software se réfère à ce champ
fichier_modele Le fichier modèle en excel Le software remplis la partie "nom" nommée "data" par une tableau comportant 3 colonnes:
1ère colonne: Numéros entiers incrémentales
2ème colonne: La liste (dans l'ordre ascendant) des numéros du produit emballées
3ème colonne: Le text "Pass"
chemin_Enr Le chemin où le fichier doit être enregistré. Ce champ est surtout utile pour le développement.
imprimante Le nom de l'imprimante où le fichier doit être imprimé Exemples:
PDFCreator,
\\CHPALSPRINT01\CHPALPB0018

Les fichiers modèle se trouve sur le réseau au chemin suivant: "U:\MTD_PRD_TESTS\PARAM_TESTS\EMBALLAGE"

Rajouter une imprimante

Il y a une table sqlite sur le réseau répertoriant les imprimantes utiles à cette application.
Cette table se trouve sur le réseaux au chemin "U:\MTD_PRD_TESTS\PARAM_TESTS\EMBALLAGE".
Cette table comporte 2 champs, le nom de l'imprimante et sont adresse IP.

Installation

Pour le bon fonctionnenement du software, le PC doit être pourvu de Microsoft Office (Word et Excel).
Ces logiciels servent pour la création des fichiers "numéro de série".

Après la première installation, il faudra rechercher le port de communication de lecteur de code barre et définir l'imprimante par défaut.
Pour modifier ces paramètres, il faut aller dans l'onglet maintenance.

Paramétrer le lecteur (Gryphon GFS4470)

Le lecteur Grypgon GFS4470 peut être paramétrer par le logiciel "Aladin" ou en scannant les codes dans la documentation.
Vous trouvez la documentation concernant le Gryphon GFS4470 au chemin suivant: T:\Labo_PLZ\Tools\Datalogic\gryphon

Dans un premier temps, il faut sélectionner l'interface "USB_COM" et le mettre en mode "Serial On Line".
Pour que le lecteur arrive à lire les étiquettes de FLATPROP (étiquettes hautes résolutions), il faut activer le code "Code ISBT 128".
Il est possible que le lecteur ne lit pas correctement les codes barres 2D. Vérifiez que le lecteur soit bien en clavier suisse.

Astuce: Essaies de scanner les codes barres 2D perpendiculairement au lecteur.
